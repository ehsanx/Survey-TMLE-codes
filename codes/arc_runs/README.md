# ARC Enhancement Runs — survey-TMLE resubmission

Self-contained, **non-destructive** run packages for the simulation and NHANES enhancement runs identified in `Writing/comments/review-v2-plan.md` ("Plan review & enhancements (2026-06-07)" + the ARC-run menu). Each run lives in its own folder, **sources the canonical engine (`codes/*.R`, `Nhanes/R/*.R`) read-only without modifying it**, writes outputs to private subfolders (never touching the locked `results/sim_full_*.csv` or `Nhanes/nhanes_output/results/`), and supports a `SMOKE=1` quick mode.

> Generated 2026-06-07 by a multi-agent build pass. **Only R01 (and R03) have been validated locally so far — smoke-test every other run before its full submit.** Treat the rest as carefully-drafted but un-run.

## Folder layout
| What | Where |
|---|---|
| Simulation run packages | `codes/arc_runs/<RID>/` (`run.R`, `submit.slurm`, `NOTES.md`, helpers) |
| NHANES run packages | `Nhanes/arc_runs/<RID>/` |
| Per-task RDS outputs | `sim_output/arc_runs/<RID>/` and `Nhanes/nhanes_output/arc_runs/<RID>/` |
| Aggregated summaries | `results/arc/<RID>_summary.csv` |
| R08 (finish E1/E2 ladder) | already prepped as `Nhanes/nhanes_finish.slurm` (not under arc_runs/) |

## One-time setup on ARC
1. Sync to scratch and **`git pull`** (the first NHANES run failed on a stale `codes/estimators.R`):
   `cd /scratch/YOUR_ALLOCATION/GitHub/survey-tmle2 && git pull`
2. The slurm scripts load `gcc/9.4.0` + `r/4.4.0` and `R_LIBS=$HOME/R/x86_64-pc-linux-gnu-library/4.4/` (the proven `codes/sim.slurm` env). The R library must have: SuperLearner, tmle, survey, surveyCV, earth, gam, glmnet, ranger, mice.
3. `mkdir -p sim_output/logs` (scripts also do this).

## Workflow for every run
1. **Smoke** (each `NOTES.md` has the exact command; `SMOKE=1`, ~1–4 min) — check the printed smoke-gate verdict.
2. **Submit** from repo root (= `SLURM_SUBMIT_DIR`): `sbatch codes/arc_runs/<RID>/submit.slurm`. Edit `--array`, `SIM_N_REPS`, `SIM_CHUNK` for TEST vs FULL per the script comments.
3. **Aggregate**: run the run's `aggregate.R` -> `results/arc/<RID>_summary.csv`.

## Run order if ARC resources are tight
| Tier | Runs (in order) | Closes |
|---|---|---|
| **1 — must-run** | R01 -> R03 -> R08 | smooth-rung bias blind spot; estimator-definition confound; completes the real-data ladder |
| **2 — high value** | R02 -> R05 -> R06 -> R04 | Theorem-2 conservatism; de-confound real data; MI headline intervals; demonstrate the product rate |
| **3 — defense** | R09 -> R10 -> R07 -> R11 | causal identification; estimand/FPC; informative-sampling; variance corroboration |
| **4 — optional** | R12 | high-DEFF stress (cut first if compute-bound) |

Minimal panel-flipping set if you can run only a few: **R01 + R03 + R02** (+ R08, ~free).

### R01_simple_control — Correctly-specified ('simple') L1 calibration control + model_type tag

**Type:** simulation | **Cost:** Cheap. 6 cells x 1000 reps, all L1 (GLM) or L2 (glm+gam+earth) -- far lighter than the headline L4 deep-RF. 32 cores/task, 64G, <=1h walltime; most tasks finish well inside that. The spec's "cheapest, highest-value run". | **SLURM:** array = 1-60 (FULL: 6 cells x ceil(1000/100)=10 chunks). TEST: 1-6 with SIM_N_REPS=100, SIM_CHUNK=100.; time = 01:00:00; mem = 64G

**Files:** codes/arc_runs/R01_simple_control/run.R, codes/arc_runs/R01_simple_control/aggregate.R, codes/arc_runs/R01_simple_control/submit.slurm, codes/arc_runs/R01_simple_control/NOTES.md

**What it does.** Pipeline-correctness control to explain the +0.012..+0.021 cross-fit/single-fit bias at L1-L3 in the headline run (results/sim_full_summary.csv), which uses model_type='complex' (Kang-Schafer misspecified, so even L1's GLM is biased). Re-runs the SAME engine and SAME five arms on model_type='simple' (apply_L is the identity, so the parametric/smooth learners are correctly specified and bias should collapse to ~0) for L1_param (control) and L2_smooth (context), for BOTH scenarios (standard, R1), 1000 reps; ALSO re-runs L1_param under 'complex' so the matched complex/simple pair sits side by side. Grid = 2 scenarios x 3 (model_type,rung) pairs = 6 cells. Every output row is tagged with a model_type column the canonical tables lack.

**New code vs engine.** All new logic lives in this run's folder; no canonical file is edited. run.R SOURCES codes/{config,dgp,estimators,diagnostics,learners}.R unchanged and reuses make_population/draw_sample/run_estimators/deff_clust as-is. The ONLY substantive change vs codes/run_sim.R: run_sim.R hardcodes make_population(scenario, model_type='complex') and draw_sample(...,'complex'); here the cell's model_type is looped, the population is built with the MATCHING model_type, and draw_sample uses that model_type. It then tags results$model_type/diagnostics$model_type. aggregate.R replicates aggregate_sim.R's coverage/bias/se_ratio/mcse math verbatim (crit=qt(.975,max(1,df)); coverage=mean(|b-Psi|<=crit*se); bias=mean(b)-Psi; emp_sd=sd(b); se_ratio=mean(se)/sd(b); mcse_cov=sqrt(cov*(1-cov)/nreps)) but adds the model_type grouping key/column and prints an OK/FLAG smoke-gate verdict on the simple/L1 Fully-Aware rows.

STOP-and-report PRIVATELY (before writing Theorem 1/2) if any simple/L1 Fully-Aware row has coverage outside ~0.93-0.96 OR |bias| beyond ~2x MCSE of 0 -> a real pipeline finding. The 20-rep smoke is a plumbing check only (large MCSE); the statistical gate is judged on the full 1000-rep output.

### R02_largem_sweep — Fixed-L4 larger-m sweep (is the over-coverage finite-m or structural?)

**Type:** simulation | **Cost:** 40 array tasks x 32 cpus, up to 6h walltime each = up to ~7680 cpu-hours worst case; in practice far less since 3 of the 4 rungs (L1/L3/RF_shallow) are much cheaper than deep-RF and 100-rep chunks parallelize across 32 cores. Deep-RF (L4) at m_total=300 is the binding cell driving the 6h request. | **SLURM:** array = 1-40; time = 06:00:00; mem = 64G

**Files:** codes/arc_runs/R02_largem_sweep/run.R, codes/arc_runs/R02_largem_sweep/helpers.R, codes/arc_runs/R02_largem_sweep/aggregate.R, codes/arc_runs/R02_largem_sweep/submit.slurm, codes/arc_runs/R02_largem_sweep/NOTES.md

**What it does.** Sweeps the number of sampled PSUs-per-stratum m=base_m across {6,12,20,30} (= baseline, 2x, ~3.3x, 5x; m_total=H*base_m={60,120,200,300}) on the FIXED 'standard' population/DGP, to settle whether the L4 cross-fit (Fully-Aware-CF) over-coverage in results/sim_full_summary.csv (se_ratio~1.44, coverage 0.985-0.992) shrinks as m grows (finite-m, conservative-consistent per Theorem 2) or is structural. 1000 reps per m. Each task runs four rungs at its m: L4_aggressive (SL.ranger.deep, the over-coverer), RF_shallow (SL.ranger.t1, the contrast whose se_ratio should approach 1 faster), and L1_param/L3_adaptive (bias track for the O(1/m) Hajek-ratio explanation of the R01 smooth-rung bias). Reports CF se_ratio+coverage+bias+MCSE vs m, plus Fully-Aware bias vs m at L1/L3.

**New code vs engine.** All new logic is in this run's folder; NO canonical file (codes/*.R) is edited. helpers.R adds: (1) one_rep_m() — mirrors run_sim.R::one_rep but forwards the m knob to draw_sample() via its EXISTING base_m argument (stage-1 SRS-WOR PSU count), holding population geometry (H=10, J_per_stratum=60, M_per_psu=200) and units-per-PSU (base_n0, default 25) fixed so only the # independent PSUs / design df = H*(base_m-1) changes; (2) summarise_sweep() — per (base_m,rung,method) summary computing bias=mean(b)-Psi, emp_sd=sd(b), se_ratio=mean(se)/emp_sd, coverage with t-df, mcse_cov=sqrt(cov(1-cov)/nreps), matching aggregate_sim.R columns and adding base_m/m_total/mean_df. run.R is the array driver (sources the canonical engine read-only, builds population ONCE, runs all 4 rungs per task across cores, writes per-task RDS). aggregate.R combines per-task RDS into the summary. The m knob reuses draw_sample(base_m=...) — codes/dgp.R is untouched.

**Smoke test:**

**Submit:**

**Expected output.** Per-task RDS -> sim_output/arc_runs/R02_largem_sweep/sim_m<NN>_chunk<CCC>.rds (+ manifest/ subdir). After aggregate.R: results/arc/R02_largem_sweep_summary.csv (one row per base_m x rung x method; columns scenario,rung,base_m,m_total,method,n_reps,Psi,bias,emp_sd,mean_se,se_ratio,coverage,mcse_cov,mean_df,deff_clust,icc_eif), results/arc/R02_largem_sweep_diag.csv (targeting/positivity per cell), and sim_output/arc_runs/R02_largem_sweep/R02_combined.rds. KEY NUMBERS TO INSPECT: Fully-Aware-CF se_ratio vs m_total for L4_aggressive (monotone decrease toward 1 => finite-m/Theorem-2; plateau >1 => structural), with coverage staying >=0.95; RF_shallow CF se_ratio should decay toward 1 faster than deep-RF; Fully-Aware bias at L1_param/L3_adaptive should shrink ~1/m. SANITY: mean_df ~ H*(base_m-1) = {50,110,190,290}. Smoke validated: df_design=290 at m=30, n~7590, w_cv~1.41, all b/se finite, 4 methods (no CV arm for single-learner L1).

Feeds a se_ratio-vs-m_total line plot (and a bias-vs-m plot) for the manuscript's Theorem-2 conservatism figure.

**Dependencies.** R 4.4 (ARC) / 4.5 (dev); packages SuperLearner, tmle, survey, surveyCV, ranger, earth, gam, glmnet (same set as run_sim.R; no new packages). **Caveats & smoke-gate.** ASSUMPTIONS/VERIFY: (1) The m knob is SAMPLED PSUs/stratum (draw_sample base_m), not population PSUs — the population and truth Psi are identical across all m; only the design's PSU count and df change (exactly the Theorem-2 m->inf regime with fixed cluster size). (2) base_m<=J_per_stratum=60, so base_m=30 is the largest valid 5x level; going beyond would require rebuilding the population with more PSUs (a different study). (3) Run is 'standard' scenario only; R1 is intrinsically m_h=2 PSUs/stratum (structurally thin df) and is not swept. SMOKE-GATE DECISION RULE: STOP-and-report (do NOT submit) if the Tier-1 smoke errors, gives non-finite b/se, or df_design != 290 at m=30 (means the base_m knob isn't changing the design dimension — premise broken). PROCEED to submit if Tier-1 passes. SCIENCE DECISION (after full run): if deep-RF Fully-Aware-CF se_ratio at m_total=300 is meaningfully below baseline 1.44 (<=~1.25) and RF_shallow is closer to 1 -> keep 'se_ratio -> 1' (finite-m/Theorem-2); if it is still >1.3 at m=300 -> pivot the manuscript claim to 'bounded valid conservatism, decreasing in m'. NOTE on local smoke: a 16GB laptop can OOM-kill PSOCK workers if many concurrent deep-RF fits run — the smoke caps cores=2 to avoid this; ARC's 64G x 32 has no such issue. If any ARC task hits the 6h walltime, halve SIM_CHUNK to 50 and double the array to 1-80.

### R03_isolation_2x2 — Isolating cross-fitting vs de-weighting (2x2) at a harmonized 0.05 floor

**Type:** simulation | **Cost:** 20 array tasks x 32 cpus x up to 4h. L4_aggressive (deep RF, 4 arms x ~2-4 cross-fit folds + 2 single fits) is the expensive rung; L1_param (GLM) is cheap. Realistic wall time per task well under the 4h cap at 32 cpus/100 reps; total ~ a few hundred core-hours, dominated by the 10 L4 tasks. | **SLURM:** array = 1-20 (standard x {L4_aggressive, L1_param} x 10 chunks of 100 reps; tasks 1-10 = L4_aggressive, 11-20 = L1_param); time = 04:00:00; mem = 64G

**Files:** codes/arc_runs/R03_isolation_2x2/estimators_isolation.R, codes/arc_runs/R03_isolation_2x2/run.R, codes/arc_runs/R03_isolation_2x2/aggregate.R, codes/arc_runs/R03_isolation_2x2/submit.slurm, codes/arc_runs/R03_isolation_2x2/NOTES.md

**What it does.** Proves CROSS-FITTING (not de-weighting) drives the L4 coverage rescue by crossing the two factors the locked Fully-Aware-vs-CF contrast confounds, at a single harmonized propensity floor g_floor=0.05. Runs FOUR arms per replication on the standard-scenario complex DGP: SF-W (single-fit/weighted-nuisance), SF-U (single-fit/UNWEIGHTED-nuisance = the dangerous tail), CF-W (cross-fit/weighted-OOF), CF-U (cross-fit/UNWEIGHTED-OOF). Two rungs: L4_aggressive (SL.ranger.deep, the demonstration) and L1_param (SL.glm, placebo where all four should cover). 1000 reps/rung. Reports bias/emp_sd/mean_se/se_ratio/coverage/mcse_cov per arm, matching the engine's column conventions.

**New code vs engine.** All new logic lives in codes/arc_runs/R03_isolation_2x2/estimators_isolation.R; no canonical file is edited. It defines run_isolation(obs, learners, g_floor=0.05, V_cf=5L) which factors the engine's nuisance estimation into a clean 2x2. .fit_nuisances() toggles two levers only: crossfit (FALSE = one Q-fit+g-fit on the full sample, predicted IN-SAMPLE; TRUE = out-of-fold fits over make_cf_folds PSU folds) and weighted (TRUE = pass obsWeights=w to .sl; FALSE = .sl(weights=NULL)). Every arm floors g at g_floor in BOTH the OOF/in-sample predictions AND the EIF reconstruction (.eif_from_tmle(..., gbound=g_floor)), removing the 1e-3-vs-0.05 asymmetry the locked engine bakes in. .one_arm() then hands the pre-computed (Q,g1W) to tmle() for the SAME weighted pooled targeting the engine's CV/CF arms use, reconstructs the EIF, and computes the clustered design SE via .se_des(). Reuses the exposed building blocks .sl / .eif_from_tmle / .se_des / make_cf_folds. run.R restricts the grid to standard x {L4_aggressive,L1_param} and adds a SMOKE path + smoke-gate verdict; aggregate.R reuses aggregate_sim.R's coverage/se_ratio/MCSE math and column names.

PRE-STATED EXPECTATION: single-fit (either weighting) under-covers at L4; cross-fit (either weighting) recovers ~nominal -> cross-fitting is the active ingredient. VERIFICATION STATUS: the run_isolation helper was validated end-to-end locally on one sample (all 4 arms produce correct method/b/se/df rows, drow diagnostics with V_eff=5 for CF and NA for SF, and a length-n EIF); the full parallel SMOKE driver (makeCluster/parLapply/clusterSetRNGStream + the L4 deep-RF path + smoke-gate block) was launched and confirmed running across many R workers but its 50-rep verdict had not finished printing at hand-off (Rscript buffers stdout to the console until exit) — re-run the documented smoke command to read the gate verdict before submitting. CAVEAT: the harmonized 0.05 floor means SF-W is NOT byte-identical to the locked Fully-Aware (which floors the EIF at 1e-3); a small mismatch is expected and intentional. CF fold assignments are randomized within the seeded RNG stream, so per-rep folds differ from the locked CF arm though the run is reproducible.

### R04_nuisance_rate — Nuisance-rate diagnostic ladder (demonstrate the product rate)

**Type:** simulation | **Cost:** 32 tasks x 32 cpus. Per-task walltime dominated by the rung: L1/L2 cheap (minutes), L4 deep-RF heaviest. 200 reps x 50/chunk = 4 chunks/cell; 4 h/task is a safe ceiling (L4 deep-RF with 5 CF folds is the binding rung).
